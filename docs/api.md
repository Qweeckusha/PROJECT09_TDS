# TDS API
> `docs-ver: 1.0` | `Framework: FastAPI`

## Введение
Данный документ описывает архитектуру внешнего API (Out-API) и его взаимодействие с внутренним ядром (API-Core). 

Система оперирует двумя основными понятиями конвейеров обработки данных:
- **`StdPl` (Standard Pipeline)** — стандартный конвейер. Задействует модули транскрибирования и диаризации, а также агрегирует результаты в единый текст с временными метками (`time`) и метками спикеров (`speaker`).
- **`FPl` (Full Pipeline)** — полный конвейер. Включает в себя `StdPl` и добавляет этап суммаризации через LLM (`StdPl + LLM`).

## Стек технологий
- **FastAPI** — фреймворк для построения API. Обеспечивает асинхронную обработку запросов и автоматическую генерацию OpenAPI (Swagger) документации.
- **Pydantic** — валидация данных и управление схемами запросов/ответов.
- **httpx** — асинхронный HTTP-клиент для скачивания файлов.


## Схемы данных (Pydantic Models)

### Запросы (Requests)
**URLRequest** (для обработки по ссылке)
- `audio_url` *(string, URL)* — прямая ссылка на аудиофайл.

**PathRequest** (для локальной обработки)
- `audio_path` *(string)* — локальный путь к аудиофайлу на сервере/в контейнере.

### Ответы (Responses)
**SummarizeResponse** (универсальный ответ)
- `status` *(string)* — статус выполнения (`success`, `error`).
- `summary` *(string | null)* — итоговый текст (транскрибация или саммари).


## Внешний контур (Out-API)

Эндпоинты разделены на логические группы: основные, тестовые и системные.

### Production Endpoints

#### `POST /summarize`
Основной эндпоинт. Принимает URL аудиофайла, скачивает его и обрабатывает через `FPl`.

**Принцип работы:**
1. **Скачивание файла:** Вызывается приватный метод `_download_file()`. Файл асинхронно скачивается чанками по 8 КБ во временную директорию.
   
   > [!NOTE]
   > *В случае таймаута или ошибки на стороне источника, возвращается **HTTP 400 Bad Request**.*

2. **Обработка (FPl):** Запускается полный конвейер.
   ```python
   try:
       summary = summarizer.get_text_from_llm(str(file_path))
       return SummarizeResponse(status="success", summary=summary)
   except Exception as e:
       raise HTTPException(status_code=500, detail=f"Ошибка обработки: {e}")
   ```
   
   > [!NOTE]
   > *Непредвиденная ошибка на этапе транскрибации, диаризации или LLM вернет **HTTP 500 Internal Server Error**.*
3. **Очистка:** В блоке `finally` временный файл удаляется из кэша, чтобы не засорять диск.

*(future) `POST /transcribe`*
Планируемый эндпоинт для обработки по ссылке через **`StdPl`** (только транскрибация и диаризация, без LLM). 

**Цель:** Перенести вызов LLM во внешние оркестраторы (например, n8n) для гибкого контроля над промптами и суммаризацией.


### 2. Testing / Local Endpoints
Предназначены для проверки работоспособности модулей внутри контейнера/сервера без необходимости поднимать внешний файловый хостинг.

#### `POST /summarize_path`
Обрабатывает локальный файл через **`FPl`** (с суммаризацией LLM).
- *Возвращает:* Суммаризированный LLM текст

#### `POST /full_txt_path`
Обрабатывает локальный файл через **`StdPl`**.
- *Возвращает:* "Сырой" текст с временными метками и спикерами, без обработки нейросетью.


### 3. System Endpoints

#### `GET /health`
Healthcheck. Возвращает `{"status": "ok"}`. Используется для мониторинга состояния сервиса.
