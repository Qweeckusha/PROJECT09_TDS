# Summarizer Module
> `docs-ver: 1.0` 

## Введение
Модуль `Summarizer` - это **оркестратор** всей системы. Он объединяет результаты транскрибации и диаризации в единый структурированный текст, а затем вызывает LLM для суммаризации.

### Ключевые задачи:
1. **Агрегация данных** - объединение результатов от `Transcriber` и `CustomDiarizer` по временным меткам
2. **Форматирование** - преобразование сырых данных в читаемый текст (разбор сценариев)
3. **Пайплайны** - реализация `StdPl` (стандартный) и `FPl` (полный с LLM)
4. **Кэширование** - сохранение результатов в файловую систему


## Архитектура класса

### Инициализация
```python
class Summarizer:
    def __init__(self):
        self.transcriber = Transcriber()
        self.diarizer = CustomDiarizer()
```

При создании экземпляра загружаются оба модуля (общий вызов именно там):
- `Transcriber` - модель Whisper (транскрибация)
- `CustomDiarizer` - пайплайн pyannote (диаризация)

> [!IMPORTANT]
> Модели загружаются в память один раз. Повторные вызовы методов не перезагружают их.


## Ключевой метод: `_align_segments()`

Это **самая важная функция** модуля. Она объединяет два независимых потока данных:
- **whisper_seg** - текст с временными метками от модуля транскрибирования
- **diarizer_seg** - спикеры с временными метками от модуля диаризации

### Входные данные:
```python
# whisper_seg:
[
    {"start": 0.0, "end": 5.42, "text": "Текст 1 сегмента"},
    {"start": 5.42, "end": 8.15, "text": "Текст 2 сегмента"},
    {"start": 8.15, "end": 12.3, "text": "Текст 3 сегмента"}
]

# diarizer_seg:
[
    {"start": 0.0, "end": 5.5, "speaker": "SPEAKER_00"},
    {"start": 5.5, "end": 12.3, "speaker": "SPEAKER_01"}
]
```

### Выходные данные:
```python
[
    {"start": 0.0, "end": 5.42, "text": "Текст 1 сегмента", "speaker": "SPEAKER_00"},
    {"start": 5.42, "end": 8.15, "text": "Текст 2 сегмента", "speaker": "SPEAKER_01"},
    {"start": 8.15, "end": 12.3, "text": "Текст 3 сегмента, спикер как у 2 сегмента", "speaker": "SPEAKER_01"}
]
```


## Алгоритм объединения: 4 сценария

Метод `_align_segments()` обрабатывает **4 возможных сценария** в зависимости от качества работы модулей:

### Сценарий A: Модуль транскрибирования не вернул текст
```python
if not whisper_seg:
    return []
```
**Причина:** Аудио без речи, ошибка модели, пустой файл  
**Результат:** Пустой список

> [!WARNING]
> Это критическая ошибка, которая не позволит завершить даже урезанную обработку пайплайна


### Сценарий B: Модуль диаризации не определил спикеров
```python
if not diarizer_seg:
    return [
        {**ws, "speaker": "UNKNOWN", "warning": "Diarization failed: module couldn't detect speakers"}
        for ws in whisper_seg
    ]
```
**Причина:** Ошибка pyannote, слишком короткий файл, сильный шум  
**Результат:** Текст есть, но все спикеры помечены как `UNKNOWN`

> [!NOTE]
> Это не критическая ошибка, и мы всё ещё можем передать безличный текст (без меток говорящего)


### Сценарий C: Только 1 спикер (оптимизация)

Появляется сложность: Это может быть: O(n) или O(n*m), где:

O - итоговый текст
n - сегменты транскрибирования
m - сегменты диаризации

В данном случае сложность - O(n), потому что мы рассматриваем только сегменты транскрибирования

```python
speakers = {seg["speaker"] for seg in diarizer_seg}
if len(speakers) == 1:
    speaker = next(iter(speakers))
    return [{**ws, "speaker": speaker} for ws in whisper_seg]
```
**Причина:** Монолог, интервью с одним человеком  
**Оптимизация:** Если спикер один - не нужно считать пересечения интервалов, просто присваиваем его всем сегментам  



### Сценарий D: 2+ спикеров (основной алгоритм)

Это **самый сложный и важный случай**. Алгоритм находит, какой спикер говорил в каждый момент времени, путем **пересечения временных интервалов**.

> [!NOTE]
> Этот случай будет наиболее частым

#### Визуализация проблемы:

```
Временная шкала:
0s          5s          10s         15s
|-----------|-----------|-----------|
[===WS1===]                              <- Текст: "Текст 1" (0-5s)
            [===WS2===]                  <- Текст: "Текст 2" (5-10s)
                        [===WS3===]      <- Текст: "Текст 3" (10-15s)

[=======DS1=======]                      <- Спикер: SPEAKER_00 (0-8s)
                    [=======DS2=======]  <- Спикер: SPEAKER_01 (8-15s)
```

**Задача:** Определить, кто говорил в каждом текстовом сегменте (WS1, WS2, WS3).

> [!NOTE]
> Стараемся отталкиваться именно от текста, он имеет высший приоритет и информативную ценность

#### Формула пересечения интервалов:

```python
overlap = max(0.0, min(ws["end"], ds["end"]) - max(ws["start"], ds["start"]))
```

**Разбор формулы:**
- `min(ws["end"], ds["end"])` - конец пересечения (минимум из двух концов)
- `max(ws["start"], ds["start"])` - начало пересечения (максимум из двух начал)
- Разница = длина пересечения
- `max(0.0, ...)` - если пересечения нет, возвращаем 0 (отрицательные значения невозможны)

#### Пример расчёта:

**WS1 (0-5s) vs DS1 (0-8s):**
```
overlap = min(5, 8) - max(0, 0) = 5 - 0 = 5.0
```
Пересечение: 5 секунд → SPEAKER_00

**WS2 (5-10s) vs DS1 (0-8s):**
```
overlap = min(10, 8) - max(5, 0) = 8 - 5 = 3.0
```
Пересечение: 3 секунды → SPEAKER_00

**WS2 (5-10s) vs DS2 (8-15s):**
```
overlap = min(10, 15) - max(5, 8) = 10 - 8 = 2.0
```
Пересечение: 2 секунды → SPEAKER_01

**Выбор:** WS2 больше пересекается с DS1 (3s > 2s) → SPEAKER_00

#### Фрагмент алгоритма:

```python
for ws in whisper_seg:  # Для каждого текстового сегмента
    best_speaker = "UNKNOWN"
    max_overlap = 0.0
    
    for ds in diarizer_seg:  # Сравниваем с каждым спикерским сегментом
        overlap = пересечение(ws, ds)
        
        if overlap > max_overlap:  # Если это лучшее пересечение
            max_overlap = overlap
            best_speaker = ds["speaker"]
    
    result.append({...ws, "speaker": best_speaker})
```

**Сложность:** O(n * m)


## Форматирование текста: `_format_full_text()`

После объединения сегментов метод преобразует их в читаемый текст.

### Логика определения формата:
```python
speakers = {seg.get("speaker", "UNKNOWN") for seg in align_segs}
is_monologue = len(speakers) == 1
```

### Монолог (1 спикер):
```
[0.0 - 5.42] Текст 1
[5.42 - 8.15] Текст 2
[8.15 - 12.3] Текст 3
```

### Диалог/полилог (2+ спикера):
```
[0.0 - 5.42] SPEAKER_00: Текст 1
[5.42 - 8.15] SPEAKER_01: Текст 2
[8.15 - 12.3] SPEAKER_01: Текст 3
```

> [!IMPORTANT]
> Если в аудио только один спикер, имена спикеров не выводятся - это делает текст чище.


## Пайплайны обработки

### Стандартный пайплайн (StdPl): `get_full_text_from_audio()`

```python
def get_full_text_from_audio(self, path: str) -> str:
    # 1. Транскрибирование
    whisper_seg = self.transcriber.transcribe_audio(path)
    
    # 2. Диаризация
    diarizer_seg = self.diarizer.Diarization(path)
    
    # 3. Объединение разнородных данных по временным меткам
    align = self._align_segments(whisper_seg, diarizer_seg)
    
    # 4. Форматирование
    text = self._format_full_text(align)
    
    # 5. Сохранение (опционально в config.py)
    if save_full_text_file:
        self._save_text(path, text)
    
    return text
```

**Результат:** Полный текст с временными метками и спикерами


### Полный пайплайн (FPl): `get_text_from_llm()`

```python
def get_text_from_llm(self, path: str) -> str:
    # 1. Выполняем StdPl
    audio_text = self.get_full_text_from_audio(path)
    
    # 2. Вызов LLM для суммаризации
    response = ol.chat(
        model=ollama_llm,
        messages=[
            {"role": "system", "content": prompt},
            {"role": "user", "content": audio_text}
        ]
    )
    
    llm_output = response.message.content
    
    # 3. Сохранение (опционально в config.py)
    if save_tds_file:
        self._save_text(path, llm_output, is_LLM=True)
    
    return llm_output
```

**Результат:** Суммаризированный текст с выводами и тезисами


## Кэширование: `_save_text()`

Метод сохраняет результаты в файловую систему для последующего анализа.

> [!WARNING]
> По умолчанию данная функция отключена в config.py для экономии места, нужна для тестов или будущих фичей

### Структура директорий:
```
project/
├── cache/
│   ├── full_text/          <- Результаты StdPl
│   │   └── transcribed-audio_{name}_{timestamp}.txt
│   └── tds/                <- Результаты FPl (с LLM)
│       └── TDS-audio_{name}_{timestamp}.txt
```

### Формат имени файла:
- **StdPl:** `transcribed-audio_{source_name}_{YYYYMMDD_HHMMSS}.txt`
- **FPl:** `TDS-audio_{source_name}_{YYYYMMDD_HHMMSS}.txt`

## Производительность и оптимизация

### Сложность алгоритмов:

| Метод | Сложность | Комментарий               |
|-------|-----------|---------------------------|
| `_align_segments()` (Сценарий C) | O(n) | Один спикер - оптимизация |
| `_align_segments()` (Сценарий D) | O(n * m) | Вложенные циклы           |

### Оптимизации:
1. **Сценарий C** - если спикер один, не считаем пересечения (не с чем пересекаться)
2. **Ленивая загрузка** - модели загружаются один раз при `__init__`
3. **Кэширование** - результаты сохраняются в файлы


## Ограничения и известные проблемы

> [!WARNING]
> 1. **Длинные файлы** - O(n*m) может быть медленным для файлов >1 часа с большим количеством сегментов
> 2. **Неточная диаризация** - если pyannote ошибается в определении спикеров, объединение будет некорректным
> 3. **Наложение голосов** - алгоритм не обрабатывает случаи, когда два спикера говорят одновременно
> 4. **Пустые результаты** - если оба модуля вернули пустые списки, метод вернет пустую строку

### Ключевые моменты для понимания:

1. **Почему `_align_segments()` - это O(n*m)?**
   - Для каждого текстового сегмента (n) мы проходим по всем спикерским сегментам (m)
   - Это вложенные циклы: `for ws in whisper_seg: for ds in diarizer_seg:`
   - Для файла на 1 час может быть ~1000 текстовых сегментов и ~500 спикерских → 500,000 итераций
   - На практике это быстро (микросекунды на итерацию), но для очень длинных файлов может быть медленно

2. **Почему формула пересечения работает?**
   - Два интервала [a, b] и [c, d] пересекаются, если `max(a, c) < min(b, d)`
   - Длина пересечения = `min(b, d) - max(a, c)`
   - Если пересечения нет, результат отрицательный → `max(0.0, ...)` обнуляет его

3. **Почему есть 4 сценария?**
   - Это обработка граничных случаев
   - Модули могут падать, возвращать пустые результаты, определять только 1 спикера
   - Без этих проверок система бы ломалась на неидеальных данных

4. **Почему монолог форматируется без спикеров?**
   - Если в аудио только один человек, вывод `SPEAKER_00:` перед каждой строкой - мусор
   - Автоматическое определение монолога делает текст чище

5. **Как это связано с API?**
   - `/summarize` вызывает `get_text_from_llm()` → FPl
   - `/full_txt_path` вызывает `get_full_text_from_audio()` → StdPl
   - API - это просто обертка над методами `Summarizer`
